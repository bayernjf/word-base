# 测试指南

本文档描述 word-base 项目支持的测试种类、适用场景及标准测试流程。

## 测试种类

| # | 测试类型 | 说明 | 适用场景 |
|---|---------|------|---------|
| 1 | **端到端测试 (E2E)** | 用浏览器工具模拟点击、输入、导航，验证完整用户流程 | 验证功能是否可用、登录流程、页面跳转 |
| 2 | **API 接口测试** | 用 curl 直接调用后端 API，验证请求/响应 | 后端逻辑、数据增删改查、认证授权 |
| 3 | **控制台错误监控** | 读取浏览器 Console 的 error/warn 日志 | 排查运行时错误、hydration 错误、环境变量问题 |
| 4 | **网络请求监控** | 检查浏览器发出的所有 HTTP 请求及响应 | 验证 API 是否被调用、URL 是否正确、状态码 |
| 5 | **静态类型检查** | 运行 `tsc` 或 `npm run check` | TypeScript 类型错误、接口定义不匹配 |
| 6 | **代码规范检查** | 运行 `npm run lint` | 代码风格、未使用变量、潜在 bug |
| 7 | **单元测试** | 运行 `npm test` 或 `vitest`/`jest` | 单个函数/组件的逻辑正确性 |
| 8 | **构建测试** | 运行 `npm run build` | 验证生产构建能否成功、有无打包错误 |
| 9 | **跨端兼容性测试** | 分别启动 Web/桌面端/移动端，验证各端运行 | 跨平台一致性、环境变量兼容 |
| 10 | **UI 快照测试** | 截图并对比页面渲染结果 | 视觉回归、布局验证 |
| 11 | **数据库验证** | 直接查询 Supabase 数据库 | 验证数据是否正确持久化、CRUD 结果 |

## 标准测试流程

全面测试时按以下顺序执行：

```
1. 静态检查（tsc + lint）         ← 先确保代码能编译
2. API 接口测试                   ← 后端接口逐个验证
3. E2E 功能测试                   ← 按功能模块走用户流程
   ├── 认证模块（登录/注册/登出）
   ├── 单词本模块（创建/删除/重命名）
   ├── 单词模块（添加/删除/编辑/列表）
   ├── AI 模型模块（加载/添加/编辑/删除/测试连接）
   ├── 智能句景模块（生成故事/切换/删除）
   ├── 设置模块（账户/外观/AI 自动分析）
   └── 数据同步模块（本地↔云端）
4. 控制台错误扫描                  ← 全程监控无报错
5. 构建测试                       ← 验证生产打包
```

## 成熟项目代码落地后的标准测试顺序

一个成熟的项目，代码从开发完成到上线发布，测试应按以下顺序逐层推进（从快到慢、从局部到整体、从低成本到高成本）：

### 阶段一：本地自测（tsc → lint → unit → integration）— 提交前必须通过

代码提交前必须通过，失败则不允许提交。

```
1. 静态类型检查（tsc / type check）
   └─ 确保类型正确，接口契约匹配
   └─ 命令：npm run check

2. 代码规范检查（lint）
   └─ 确保代码风格一致，无未使用变量、无潜在 bug
   └─ 命令：npm run lint

3. 单元测试（unit test）
   └─ 验证单个函数/组件/工具逻辑的正确性
   └─ 命令：npm test
   └─ 要求：覆盖率达标，新增代码必须有对应测试

4. 集成测试（integration test）
   └─ 验证多个模块组合后的行为（如 API + 数据库）
   └─ 命令：npm run test:integration（如有）
```

### 阶段二：联调测试（API → E2E → Console → Network）— 创建 PR 前必须通过

验证前后端联调、完整数据链路是否通畅。

```
5. API 接口测试
   └─ 用 curl / Postman 逐个验证后端接口
   └─ 覆盖正常路径 + 边界情况 + 异常处理
   └─ 验证项：状态码、响应结构、错误信息、权限控制

6. E2E 功能测试
   └─ 用浏览器模拟真实用户操作，走完整业务流程
   └─ 按功能模块逐一验证：
      ├── 认证模块（注册/登录/登出/会话保持）
      ├── 核心业务模块（CRUD 全流程）
      ├── AI 功能模块（模型加载/调用/错误处理）
      ├── 设置模块（配置保存/读取）
      └── 数据同步模块（本地↔云端一致性）

7. 控制台错误扫描
   └─ 全程监控浏览器 Console，确保无 error/warn
   └─ 重点关注：hydration 错误、环境变量缺失、网络请求失败

8. 网络请求监控
   └─ 检查所有 API 请求的 URL、Method、状态码、响应时间
   └─ 确保无多余请求、无重复请求、无 CORS 错误
```

### 阶段三：构建与部署测试（build → 跨端 → 预览部署）— 合并到 main 前必须通过

验证生产环境构建和部署流程。

```
9. 构建测试（build）
   └─ 验证生产打包能否成功
   └─ 命令：npm run build
   └─ 检查项：无打包错误、无未解决的依赖、产物大小合理

10. 跨端兼容性测试
    └─ 分别在 Web / 桌面端（Tauri）/ 移动端（Expo）运行
    └─ 验证各端功能一致、样式正确、环境变量读取正常

11. 预览环境部署验证
    └─ 部署到 Vercel Preview / Cloudflare Preview
    └─ 验证线上环境功能正常（环境变量、API 代理、数据库连接）
```

### 阶段四：上线前回归测试（回归 → 迁移 → 安全 → 性能）— 发布生产前必须通过

合并到 main 分支、正式发布前执行。

```
12. 回归测试（regression）
    └─ 重新执行核心 E2E 用例，确保新代码未破坏已有功能
    └─ 重点关注：本次修改涉及的功能模块 + 关联模块

13. 数据库迁移验证
    └─ 验证 migration 脚本执行无误
    └─ 检查表结构、索引、RLS 策略是否符合预期
    └─ 验证数据回填（如有）正确

14. 安全检查
    └─ 确保无密钥泄露（.env 不在 git 中、前端无敏感 key）
    └─ 验证 RLS 策略（用户只能访问自己的数据）
    └─ 验证 API 鉴权（未登录无法访问受保护接口）

15. 性能验证（可选）
    └─ 页面首屏加载时间
    └─ API 响应时间（P95 < 500ms）
    └─ 数据库慢查询检查
```

### 阶段五：上线后监控（烟雾测试 → 监控告警）— 出问题触发回滚

```
16. 烟雾测试（smoke test）
    └─ 上线后立即验证核心功能是否可用
    └─ 登录、核心 CRUD、AI 调用、数据同步

17. 监控告警
    └─ 关注错误率、响应时间、用户反馈
    └─ 如有问题，触发回滚流程（rollback.yml）
```

### 测试顺序速查

```
本地自测 → 联调测试 → 构建部署 → 上线前回归 → 上线后监控
  tsc       API          build       回归         烟雾
  lint      E2E          跨端        迁移         监控
  unit      Console      预览部署    安全
  integ     Network                  性能
```

### 各阶段的"必须通过"原则

| 阶段 | 失败后的处理 |
|------|-------------|
| 本地自测 | **禁止提交代码**，必须修复后才能 commit |
| 联调测试 | **禁止创建 PR**，必须修复后才能发起合并请求 |
| 构建部署 | **禁止合并到 main**，必须修复后才能合并 |
| 上线前回归 | **禁止发布上线**，必须修复后才能部署生产 |
| 上线后监控 | **触发回滚**，回到上一个稳定版本 |

## 指令参考

可以通过以下指令快速触发对应测试：

- **"做一次 E2E 测试"** → 用浏览器模拟用户操作走完整流程
- **"跑一下 API 测试"** → 用 curl 逐个验证后端接口
- **"做静态检查"** → 跑 tsc + lint
- **"全面测试"** → 按上面的完整流程走一遍
- **"测试 XX 功能"** → 针对指定功能做 E2E + API 验证

## E2E 测试工具链

E2E 测试依赖集成的浏览器 MCP 工具（`integrated_browser`），主要使用的工具方法：

| 工具 | 用途 |
|------|------|
| `browser_navigate` | 导航到指定 URL |
| `browser_snapshot` | 获取页面可交互元素快照 |
| `browser_click` | 点击指定元素 |
| `browser_type` | 在输入框中输入文本 |
| `browser_console_messages` | 读取浏览器控制台日志 |
| `browser_network_requests` | 查看所有网络请求 |
| `browser_take_screenshot` | 截图保存 |
| `browser_wait_for` | 等待条件满足 |

## API 测试示例

```bash
# 登录获取 token
curl -s -X POST http://localhost:3001/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"your@email.com","password":"yourpassword"}'

# 带 token 请求 AI 模型列表
curl -s http://localhost:3001/api/v1/ai/providers \
  -H "Authorization: Bearer <accessToken>"
```

## 前置条件

执行测试前需确保：

1. **API 后端已启动**：`npm run dev:api`（端口 3001）
2. **Web 前端已启动**：`npm run dev`（端口 3000）
3. **环境变量已配置**：`.env.local` 中包含 `NEXT_PUBLIC_SUPABASE_URL`、`NEXT_PUBLIC_SUPABASE_ANON_KEY`、`NEXT_PUBLIC_API_BASE_URL`
4. **Supabase 后端可达**：数据库迁移已执行，表结构已创建
5. **Node 版本正确**：使用 Node 22 LTS（见 `.node-version`）
